Ideation: Observations Self-Improvement

2026-06-19 repo-grounded Focus: observations self improvement

Grounding Context

Codebase Context

Python (≥3.11) CLI using Typer, MSAL, Pydantic, Requests, PyYAML. Read-only SharePoint CLI wrapping Microsoft Graph API with 6 approved query intents. The observations subsystem captures 4 event types (invocation, error, crash, interaction_gap) to JSONL files at ~/.local/share/workboard/observations/. An analysis script (scripts/analyze-observations.py) reads JSONL and produces a ranked markdown report. The self-improvement roadmap is drafted but mostly unimplemented.

STRATEGY.md identifies "Auto-Refinement" as an active track: "the agent monitors interaction patterns and surfaced data to proactively suggest schema updates, new query intents, stale documentation refreshes, and output improvements without human triage."

User-Named References (constraint)

docs/roadmaps/observations-self-improvement.md — 3-phase roadmap:

  • Phase 1 (Quick Wins): observations stats CLI command, stderr gap warnings on slow endpoints, keyword clustering for unsupported intents
  • Phase 2 (Closed Loop): Threshold-based gap promotion → auto-PR drafts → outcome trend dashboard → recent failures context injection
  • Phase 3 (Advanced): Semantic intent routing (embeddings), heuristic outcome inference, schema drift auto-detection, cross-session behavioral change

1 Correlation ID Across Observation Streams
Conf: 95% Cx: Low Axis: Capture depth
Description
Add a session-scoped UUID (session_id) generated once per CLI process and stamped on every observation event — both CLI events (invocation, error, crash, interaction_gap) and agent observations. Currently, CLI and agent observations land in separate JSONL files with no shared join key: the CLI uses ts + ms per event, the agent uses ts per entry. No deterministic way exists to correlate "what the CLI did" with "what happened in the conversation" for the same session. One uuid4() call at module init solves this permanently.
Basis
direct: CLI and agent write to separate files with "no cross-file correlation key" (evidence-capture-depth.md:211). The capture module already generates ts and ms per event but no session-level identifier. The agent observation file (workboard-agent-observations.jsonl) and CLI file (workboard-observations.jsonl) are unjoinable.
Rationale
Every downstream analysis improvement — intent-level error clustering, outcome-by-endpoint analysis, root-cause linking between "user kept rephrasing" and "endpoint consistently returned 504" — depends on joining CLI and agent views of the same session. Without a correlation key, the analysis script cannot answer "did the agent outcome match the CLI outcome for this session?" — the core metric for Phase 2.3's trend dashboard.
Downsides
Adds one UUID field (~36 bytes) per JSONL event. Negligible storage impact. Agent must be updated to copy the session_id from the CLI into its own observation entries.
2 CLI Command with Structured Output
Conf: 90% Cx: Low Axis: CLI surface & agent hooks
Description
Replace the standalone scripts/analyze-observations.py (markdown-only output, argparse, unregistered script path) with workboard observations stats --days N --format json|markdown — a first-class Typer command under a new observations sub-app. Structured JSON output is the primary mode for agent consumption; agent automation (Phase 2.4 context injection, Phase 2.3 trend dashboard) reads this instead of reimplementing observation directory scanning.
Basis
direct: Roadmap Phase 1.1 "Add observations stats CLI command" is unimplemented. analyze-observations.py outputs markdown only — no --format json flag. The analysis script duplicates path resolution logic from observations.py (cli-surface.md:97-104).
Rationale
Phase 2's gap promotion (2.1), trend dashboard (2.3), and context injection (2.4) all need programmatic access to observation summaries — a CLI command with --json is the stable internal API they consume. Building it first means each downstream automation step does not need to re-solve "how do I read the observations directory and parse events." Once workboard observations stats --json exists, a scheduled cron can pipe into trend aggregation, a CI check can diff week-over-week changes, and the agent can summon latest gaps at session start with a single CLI call.
Downsides
Increases CLI surface area (new subcommand group). Must handle case where no observation files exist gracefully. Analysis logic currently lives in a script — will need refactoring into a library function callable from both CLI and script contexts.
3 Session-Unique Deduplication
Conf: 85% Cx: Low Axis: Analysis & clustering
Description
Replace the raw Counter.most_common() in analyze-observations.py with session-unique grouping: count each distinct error code or unsupported intent at most once per session. If a single fat-finger session produces 15 "unknown field" errors, that session contributes 1 to the count, not 15. The roadmap explicitly mandates this principle (line 71: "Use session-unique counting, not raw count") but the analysis script actively violates it.
Basis
direct: Roadmap observations-self-improvement.md:71 "Use session-unique counting, not raw count — filter noise." analyze-observations.py:90-95 uses Counter(ev.get("code", "unknown")) with no session grouping. A single noisy session dominates the report, hiding real multi-session patterns.
Rationale
The analysis script is the gate through which all observation data flows. If it misrepresents the data by amplifying single-session noise, every downstream decision — which gap to fix, which intent to add, which endpoint to optimize — is made on misleading information. This directly contradicts the roadmap's own noise-filtering principle and must be fixed before any automated gap promotion can trust the analysis output.
Downsides
Requires session IDs on events (see idea #1). Without correlation IDs, session grouping must fall back to approximate heuristics (same ts sliding window). Temporary heuristic acceptable while #1 is implemented.
4 get_all() Pagination Instrumentation
Conf: 95% Cx: Very Low Axis: Capture depth
Description
Move the on_gap callback invocation from get() into the shared HTTP execution path so get_all() — which paginates through multiple result pages — fires the callback on every request. Additionally, aggregate the total operation duration across all pagination rounds and emit a single aggregated gap event when total time exceeds the threshold. Currently, get_all() (graph_client.py:68-78) never calls _on_gap, making paginated fetches — the most expensive operations in the CLI — completely invisible to the observation system.
Basis
direct: graph_client.py:53-61 (get() calls on_gap) vs graph_client.py:68-78 (get_all() does not). The 3000ms threshold applies per-request, not per-operation. All core SharePoint data operations (list items, columns, lists) go through get_all() (sharepoint.py:17-40).
Rationale
The CLI's most expensive operations — paginated fetches for "all open items", "all overdue items" that typically return 500+ results — are invisible to the gap detection system. A paginated sequence of 20 × 500ms requests takes 10 seconds and registers zero gaps. This is a 2-line code change (moving the callback call) that surfaces the longest-running operations. Combined with stderr markers, users and agents would see gap events for pagination performance problems that currently go unmeasured.
Downsides
May produce gap events for normal pagination (every get_all will now fire per-page callbacks). Aggregated total-time gap is the right fix — per-page callbacks alone would be noisy. Implementation requires tracking cumulative duration across the pagination loop.
The gap: every request through get() fires the gap callback, but get_all() pagination bypasses it entirely.
5 Scheduled Observation Pulse in CI
Conf: 80% Cx: Low Axis: Automation & closed-loop
Description
Add a weekly-scheduled GitHub Actions workflow that runs workboard observations stats --days 7 --format markdown and posts the output as a comment on a tracking issue (or creates a new issue if a trend crosses a threshold). One workflow file (~25 lines) guarantees a weekly analysis pulse forever. This single action directly addresses the roadmap's self-identified risk: "Cadence decay — weekly review won't sustain" (observations-self-improvement.md:74).
Basis
direct: "Cadence decay — weekly review won't sustain. Phase 2 automation exists precisely because human cadence doesn't scale" (evidence-automation-loop.md:84-86). "No scheduled or CI automation exists" (evidence-automation-loop.md:68-80). Only push/PR CI workflow exists.
Rationale
The single greatest risk to the self-improvement system is "who runs the review?" A weekly cron workflow eliminates this question entirely. Once this cadence exists, every other automation step (gap promotion, trend dashboard, context injection) adds its output to the same weekly pulse instead of requiring a separate trigger mechanism. The compound effect: the weekly issue post becomes the canonical "what changed this week" artifact, and new automation features are visible to the team the week they ship.
Downsides
Requires GitHub Actions runner with environment configured (token, Python setup). Output quality depends on observations stats command being implemented first (idea #2). Weekly cadence may be too slow for fast-moving issues; a complementary on-demand command covers that case.
6 Self-Correction Capture (Retry Instrumentation)
Conf: 80% Cx: Low Axis: Automation & closed-loop
Description
Add observation capture to the existing retry-on-429 logic in graph_client.py (lines 62-64). Log each retry attempt as a structured observation with endpoint, attempt number, delay, and success/failure. Aggregate across sessions to detect persistent hot endpoints. When an endpoint exceeds N retries across M sessions, suggest or auto-adjust the retry delay or cache TTL for that path. This is a concrete, low-risk step toward Phase 3.4 cross-session behavioral change.
Basis
direct: Retry logic exists at graph_client.py:62-64 but captures nothing. "A retry is self-correction. A persistent behavioral change across sessions is self-improvement" (evidence-automation-loop.md:166). Roadmap Phase 3.4 targets cross-session behavior change.
Rationale
This bridges the self-correction / self-improvement gap with a concrete mechanism that costs ~5 lines of code. The retry loop already exists; adding observation capture is trivial. Aggregation across sessions is a small extension to the analysis pipeline. The payoff is that throttle behavior improves without human intervention — each session's failures make the next session's behavior smarter. This is the simplest version of "the system detects, diagnoses, and fixes its own gaps" and proves the pattern in a low-risk domain before applying it to intent routing or schema drift.
Downsides
Increases observation volume modestly (each retry sequence adds a few events). Only meaningful if aggregated across sessions — standalone events have limited value. Requires naming convention for throttle-related observations to avoid noise in standard error analysis.
7 Iterative Error Clustering (n-gram → tf-idf → embeddings)
Conf: 75% Cx: Medium Axis: Analysis & clustering
Description
Start with simple n-gram / token-overlap clustering on the detail and code fields of unsupported_intent errors (Roadmap Phase 1.3). Cluster output feeds back as suggested new intent entries for agent-intents.md. Each subsequent iteration upgrades the clusterer: bigger n-grams → tf-idf → embeddings (Phase 3.1) — while keeping the same pipeline inputs and output format. The first version (n-gram) ranks candidate intents and establishes the input/output contract; the second (tf-idf) produces cleaner clusters with centroid-term names; the third (embeddings) routes real queries automatically.
Basis
direct: Roadmap Phase 1.3 "keyword clustering to analysis script" (observations-self-improvement.md:24). Roadmap Phase 3.1 "semantic intent routing with embeddings" (observations-self-improvement.md:43). Current code: strict APPROVED_INTENTS set match with no fallback (evidence-analysis-clustering.md passim).
Rationale
The compounding effect is in the pipeline architecture — the first version (n-gram) gives you ranked candidate intents and establishes the input/output contract. Each upgrade costs less than the one before because the data pipeline, test harness, and evaluation methodology already exist. Without this iterative approach, there is a temptation to skip straight to embeddings — which requires labeled data that does not exist yet. The n-gram output becomes that labeled training data.
Downsides
n-gram clustering produces noisy clusters initially (synonyms, minor phrasing differences). Requires human review of suggested intents before promotion — cannot be fully automated at the n-gram stage. Medium complexity because it involves new Python dependencies and an output format that feeds back into the agent intents workflow.
